# Favornoms — Full Project Audit (2026-06-24)

White-label restaurant **delivery** SaaS. Platform owner rents the system to restaurant
owners as a package; each restaurant runs multiple branches, each branch has its own
storefront + QR, isolated loyalty/identity between restaurants, and a customizable menu
layout.

> **Method:** static exploration of all 5 apps + `packages/*` + `supabase/*`, a competitor
> sweep (Flipdish / ChowNow / Olo / Toast / Ordering.co), and a live read of the Supabase
> **security + performance advisors** (114 security findings, 231 performance findings).
> No code or DB was changed in this audit pass.

---

## 0. TL;DR

The product is **far more complete than a typical MVP** — multi-tenant routing, custom
domains, full checkout, live delivery tracking + chat, driver dispatch v2.1, loyalty,
hours/pause/surge, an onboarding wizard, plan tiers, and a platform-owner dashboard are all
built and (per prior smoke tests) working. The work that separates it from "shippable as a
paid product" is concentrated in **three areas**: (1) the **money layer is not wired**,
(2) a handful of **real security holes** the linter caught, and (3) the **loyalty/identity
model in code doesn't yet match the owner's decision** (shared across a restaurant's
branches).

## 1. Locked decisions (2026-06-24) and the gaps they create

The owner locked three previously-deferred decisions this session:

### D1 — Market = United States only
Keep US assumptions (sales tax, Stripe, Twilio SMS, email push). SEA/Thai rails
(PromptPay, COD-first, LINE) are explicitly **out of scope**.
- **Gap:** onboarding wizard still ships Thai placeholder copy (`somtam-zab`, "Sukhumvit
  branch") — cosmetic, swap to US examples.

### D2 — Money model = restaurant-receives + subscription-only (NOT a marketplace)
Each restaurant collects its own order payments; the platform takes **no cut of orders** and
earns **only the merchant subscription**. → Do **not** build Stripe Connect
application-fee/destination charges.
- **Gap A (customer→restaurant):** `confirmCardPayment` is a CDN stub. Need per-restaurant
  payment connection (each restaurant's own Stripe account / keys) so funds land in the
  restaurant's account. Cash/pay-at-counter already works.
- **Gap B (restaurant→platform):** the "Upgrade plan" button only flips a DB row — the code
  literally says *"Stripe billing not yet wired."* Needs real **Stripe Billing / Checkout
  subscription** so the platform actually collects rent.

### D3 — Loyalty + customer identity shared across ALL branches of a restaurant (scope=`brand`)
Isolated **between** restaurants (already true), shared **within** a restaurant's branches.
- **Gap:** code is per-**branch** today: `customers.branch_id` is required (one row per
  branch), and `listMyLoyaltyTransactions` filters by `branch_id`
  (`packages/database/src/queries/loyalty.ts:56`). `get_loyalty_balance` already honors
  `brand` scope, but identity resolution + transaction history do not. Need: default
  `loyalty_scope='brand'`, restaurant-level customer resolution, and brand-wide history.

## 2. Findings by severity

### 🔴 CRITICAL

**C1 — Two public tables have RLS disabled → cross-tenant data exposure**
Supabase security advisor (ERROR): `public.birthday_rewards` and
`public.referral_redemptions` are exposed via PostgREST with **no RLS enabled**. Any client
holding the anon/publishable key can read (and likely write) every tenant's rows. This
directly violates the "separate restaurants" promise.
*Fix:* `ALTER TABLE … ENABLE ROW LEVEL SECURITY;` + add branch/restaurant-scoped policies
matching sibling loyalty tables. — https://supabase.com/docs/guides/database/database-linter

**C2 — Money flow not wired (both halves)** — see D2. No real revenue can be collected
today (orders fall back to cash; plan upgrades don't charge). This is the #1 product blocker.

**C3 — Loyalty/identity model contradicts the locked decision** — see D3. Shipping as-is
means a customer's points/history fragment per branch, which the owner explicitly does not
want.

### 🟠 HIGH

**H1 — Privileged `SECURITY DEFINER` RPCs callable by `anon`** (advisor: 30 functions).
Most are trigger helpers, but some look privileged — e.g. `admin_edit_order_notes` is
listed as executable by the `anon` role. If these don't re-check the caller's role
internally, that's an auth-bypass. *Fix:* `REVOKE EXECUTE … FROM anon` on anything not meant
for guests; audit each for internal authorization. (79 more are open to `authenticated` —
expected for an RPC app, but each should self-authorize.)

**H2 — Cart shows a bogus `$40` delivery fee + 5% service fee, mismatching checkout.**
`cart-view.tsx:42-43` hardcodes `deliveryFee = 40` and `serviceFee = subtotal*0.05`. The
`40` is a leftover from the old THB build (40฿); in a USD app this reads as **$40**. Checkout
computes a real distance-based quote, so the two screens disagree. *Fix:* show "calculated at
checkout" or reuse the live `quote_delivery` result; drop the invented service fee unless
intended.

**H3 — Substantial finished work sits uncommitted** in the working tree (account
pages: addresses/loyalty/settings; `packages/maps` location-picker/geolocation/reverse-
geocode; several modified files). It's outside git history and CI — risk of loss and no
build verification. *Fix:* review, fix H2, then commit. (This is one of the two things you
asked me to do next.)

**H4 — Branding is limited to 2 colors; no per-restaurant logo / hero image.** Hero image is
a hardcoded Unsplash URL. For a white-label product this is a core expectation. *Fix:* add
logo + hero upload (Supabase Storage) to branch/brand settings, apply on the storefront.

### 🟡 MEDIUM

- **M1 — `auth_rls_initplan` (44 policies):** RLS calls `auth.uid()`/`current_setting()`
  per-row instead of `(select auth.uid())`. Known Supabase perf footgun at scale. Easy
  rewrite. — performance advisor.
- **M2 — 38 foreign keys without a covering index** → slow joins / cascade deletes as data
  grows. Add indexes.
- **M3 — 110 "multiple permissive policies"** (same role+action). Consolidate to cut
  per-query policy evaluation cost.
- **M4 — Dev auth backdoors** (`apps/admin` + `apps/driver` `…/api/test-sign-in`) must be
  removed or env-gated before production.
- **M5 — Tier-3 competitive gaps** (vs Flipdish/ChowNow): abandoned-cart recovery
  (table+cron exist — verify it's wired E2E), email/SMS marketing campaigns + segmentation,
  upsell/cross-sell engine, third-party DSP fallback (DoorDash Drive / Uber Direct).
- **M6 — `tier_for_lifetime_points` has a mutable `search_path`** (set it explicitly).

### 🟢 LOW

- **L1 — Voice-order button is a stub** (cart). Hide or implement.
- **L2 — 39 unused indexes** (drop later to save write cost; may just be low traffic).
- **L3 — `rate_limits` has RLS enabled but no policy** (locks the table; fine if only touched
  via SECURITY DEFINER — confirm).
- **L4 — Leaked-password protection off** (minor; auth is OTP/magic-link first).
- **L5 — Thai placeholder copy** in onboarding (D1).

## 3. Security & multi-tenancy summary (advisor-backed)

| Finding | Level | Count |
|---|---|---|
| `rls_disabled_in_public` (birthday_rewards, referral_redemptions) | **ERROR** | 2 |
| `anon_security_definer_function_executable` | WARN | 30 |
| `authenticated_security_definer_function_executable` | WARN | 79 |
| `function_search_path_mutable` | WARN | 1 |
| `auth_leaked_password_protection` | WARN | 1 |
| `rls_enabled_no_policy` (rate_limits) | INFO | 1 |

The tenant model itself is sound (shared DB + RLS via `user_branch_ids()`, public read on
active branches, customer-owned rows). The issues above are **specific holes**, not a broken
design — all fixable with a focused migration + a `REVOKE`/policy pass.

## 4. Performance summary (advisor-backed)

231 findings, none fatal, all "fix before scale": 110 multiple-permissive-policies,
44 `auth_rls_initplan`, 39 unused indexes, 38 unindexed FKs. A single cleanup migration plus
the `(select auth.uid())` rewrite clears the bulk.

## 5. Recommended roadmap (aligned to the locked decisions)

   campaigns → DSP fallback (M5).
